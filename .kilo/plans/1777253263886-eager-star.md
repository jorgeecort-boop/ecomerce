## 🔍 FASE 1: DIAGNÓSTICO DE SÓLO LECTURA

### 1️⃣ CAPA DE DATOS (Prisma/NestJS) - Análisis de Rendimiento

#### 🚨 Problemas Críticos Identificados:

**A. Consultas N+1 y Sobre-fetching en `OrdersService`**
- **Archivo:** `apps/api/src/modules/orders/orders.service.ts`
- **Línea 96-104:** `findAllByStore` incluye `items.product` completo
  ```typescript
  product: { select: { id: true, title: true, images: true } }
  ```
  **Impacto:** Cada ítem de orden carga el producto completo (incluyendo `images` como JSON string). En listados de dashboard con 100+ órdenes, esto genera ~5-10MB de transferencia innecesaria.
  **Patrón N+1:** Si se accede a propiedades anidadas del producto en el frontend, genera queries adicionales.

**B. Falta de Paginación Efectiva en `DashboardUnifiedService`**
- **Archivo:** `apps/api/src/modules/dashboard-unified/dashboard-unified.service.ts`
- **Línea 95-100:** `findMany` con `take: 50` sin cursor-based pagination
  ```typescript
  take: 50, // Hardcoded, no offset/after cursor
  ```
  **Impacto:** Para tiendas con 10k+ órdenes, el rendimiento decae linealmente. No hay `skip` dinámico.

**C. Consultas Secuenciales en `DashboardService`**
- **Archivo:** `apps/api/src/modules/dashboard/dashboard.service.ts`
- **Línea 23-50:** 5 queries secuenciales con `Promise.all` pero sin índices optimizados
  ```typescript
  const [orders, products, totalRevenue, totalOrders, lowStock] = await Promise.all([...])
  ```
  **Impacto:** Cada query hace full table scan. Faltan índices compuestos en:
  - `Order(storeId, paymentStatus, createdAt)`
  - `Product(storeId, trackInventory, inventory)`

**D. Problemas de Consistencia en `CartService`**
- **Archivo:** `apps/api/src/modules/cart/cart.service.ts`
- **Línea 220-229:** Actualización de inventario fuera de transacción atómica
  ```typescript
  for (const item of cart.items) {
    await this.prisma.product.update({ ... }) // ❌ No está en $transaction
  }
  ```
  **Impacto:** Race condition: dos compras simultáneas pueden sobre-vender stock.

**E. Falta de Selectividad en `ProductsService.findPublishedByStore`**
- **Archivo:** `apps/api/src/modules/products/products.service.ts`
- **Línea 52-67:** `select` incluye `images` (JSON grande) innecesario para catálogo público
  ```typescript
  images: true, // Carga todo el array JSON
  ```
  **Impacto:** Respuestas HTTP de 200-500KB por producto en lugar de 2-5KB.

### 2️⃣ CAPA DE LÓGICA (NestJS) - Acoplamiento y Errores

#### 🚨 Problemas Críticos Identificados:

**A. Acoplamiento Fuerte: `DashboardUnifiedService`**
- **Archivo:** `apps/api/src/modules/dashboard-unified/dashboard-unified.service.ts`
- **Línea 1-12:** Inyección directa de `ShopifyService` y `AutoFulfillmentService`
  ```typescript
  constructor(
    private prisma: PrismaService,
    private shopifyService: ShopifyService,  // ❌ Acoplamiento directo
    private autoFulfillmentService: AutoFulfillmentService
  )
  ```
  **Impacto:** Viola principio de DIP (Dependency Inversion). Cambios en Shopify rompen el dashboard unificado. Difícil de testear (requiere mocks complejos).

**B. Duplicación de Lógica: Manejo de Tiendas**
- **Patrón repetido** en `DashboardService`, `DashboardUnifiedService`, `OrdersService`:
  ```typescript
  const stores = await this.prisma.store.findMany({ where: { ownerId: userId } });
  const storeIds = stores.map(s => s.id);
  ```
  **Impacto:** 3+ lugares con misma lógica. Si cambia el modelo, hay que actualizar N archivos.

**C. Falta de Exception Filters Centralizados**
- **Archivo:** No existe `apps/api/src/common/filters/`
- **Impacto:** Cada controlador maneja errores manualmente:
  ```typescript
  // En 15+ controladores:
  if (!entity) throw new NotFoundException('...');
  ```
  **Patrón inconsistente:** Algunos devuelven `NotFoundException`, otros `BadRequestException` para mismo caso.

**D. Código Espagueti en `CartService.checkout`**
- **Archivo:** `apps/api/src/modules/cart/cart.service.ts`
- **Línea 165-240:** 75 líneas haciendo:
  1. Validación
  2. Creación orden
  3. Actualización inventario
  4. Notificación Telegram
  5. Limpieza carrito
  
  **Impacto:** Viola SRP (Single Responsibility Principle). Difícil de testear, mantener y extender.

**E. Manejo de Errores Asíncronos Inconsistente**
- **Archivo:** `apps/api/src/modules/shopify/shopify.service.ts`
- **Línea 85-92:** Silencia errores devolviendo `[]`
  ```typescript
  } catch (error) {
    this.logger.error('Failed to fetch orders', error);
    return []; // ❌ El consumidor no sabe si falló
  }
  ```
  **Impacto:** Comportamiento indefinido en el frontend (muestra "0 productos" en lugar de "Error de conexión").

### 3️⃣ CAPA DE PRESENTACIÓN (Next.js) - Renderizado y Código Muerto

#### 🚨 Problemas Críticos Identificados:

**A. Uso Excesivo de `"use client"`**
- **Archivo:** `apps/web/src/app/store/[slug]/StoreClient.tsx`
- **Línea 1:** Todo el componente es cliente, incluyendo:
  - Renderizado de lista de productos (no interactivo)
  - Mostrar precios (no interactivo)
  
  **Impacto:** Hydration innecesaria, aumento de bundle (~150KB), FCP más lento.

**B. Componentes Interactivos No Aislados**
- **Patrón:** Botones "Add to Cart" dentro de listas renderizadas en servidor deberían ser client components pequeños, no todo el contenedor.
- **Ejemplo:** `StoreClient.tsx` líneas 158-212: Todo el `<div className="grid grid-cols-1...">` es cliente.

**C. Dependencias Muertas / No Utilizadas**
- **Análisis de imports no utilizados:**
  - `apps/web/src/components/features/home/Hero.tsx`: Importa `useRouter` pero no lo usa
  - `apps/web/src/components/features/home/CategoryGrid.tsx`: Importa `useEffect` innecesario
  - `apps/api/src/modules/currency/currency.controller.ts`: Importa `Query` pero no lo usa

**D. Faltan Server Components para SEO**
- **Archivo:** `apps/web/src/app/store/[slug]/page.tsx`
- **Estado actual:** ✅ Ya está corregido (Server Component)
- **Pero:** `StoreClient.tsx` sigue siendo "use client" completo

**E. Código Duplicado en Formatters**
- **Patrón:** `formatCurrency` implementado en:
  1. `apps/web/src/hooks/useCurrency.ts` (Intl.NumberFormat)
  2. `apps/web/src/app/store/[slug]/checkout/page.tsx` (función local `formatCurrency`)
  
  **Impacto:** Inconsistencia en formato (COP vs USD display).

**F. Hydration Mismatches Potenciales**
- **Archivo:** `apps/web/src/hooks/useCurrency.ts`
- **Línea 118:** `if (!mounted) return `${symbol}${converted.toFixed(2)}`;`
- **Línea 121-126:** Usa `Intl.NumberFormat` solo en cliente
  
  **Impacto:** El servidor renderiza "USD10.00", cliente renderiza "$10.00" → Hydration warning.

## 🎯 FASE 2: PLAN DE EJECUCIÓN POR FASES

### Prioridad 1: CRÍTICO - Consistencia de Datos y Race Conditions

#### ✅ Componente A: `CartService` - Transacciones Atómicas
**Razón:** Race condition en inventario puede causar sobre-venta (pérdida financiera directa).

**Cambios:**
1. Mover actualización de inventario dentro de `prisma.$transaction` en `createGuestOrder`
2. Aplicar mismo patrón en `CartService.checkout`
3. Agregar `skipLocked` para colas de inventario

**Tests Requeridos:**
- Test de concurrencia: 10 requests simultáneos comprando mismo producto (stock: 5)
- Verificar que solo 5 órdenes se crean con éxito
- Verificar que el stock final es 0

#### ✅ Componente B: `DashboardUnifiedService` - DIP y Desacoplamiento
**Razón:** Alto acoplamiento hace el sistema frágil a cambios.

**Cambios:**
1. Crear interfaces (`IShopifyService`, `IAutoFulfillmentService`)
2. Inyectar dependencias vía contratos, no implementaciones
3. Extraer lógica de merge de datos a servicio separado

**Tests Requeridos:**
- Test unitario con mocks de interfaces
- Verificar que Dashboard funciona sin Shopify conectado

---

### Prioridad 2: ALTO - Rendimiento y Escalabilidad

#### ✅ Componente C: `OrdersService` - Optimización Queries
**Razón:** N+1 queries y sobre-fetching matan rendimiento en listados grandes.

**Cambios:**
1. En `findAllByStore`: Remover `product` del include, usar solo datos planos
2. Agregar índices Prisma:
   ```prisma
   @@index([storeId, createdAt])
   @@index([paymentStatus, storeId])
   ```
3. Implementar cursor-based pagination (Relay-style)

**Tests Requeridos:**
- Benchmark: 10k órdenes, medir tiempo de query y memoria
- Test de paginación: verificar `hasNextPage`, `endCursor`

#### ✅ Componente D: `ProductsService` - Selectividad
**Razón:** Catálogo público carga JSON de imágenes innecesariamente.

**Cambios:**
1. Separar `findPublishedByStore` en dos métodos:
   - `findPublishedSummary` (para catálogo: id, title, price, slug)
   - `findPublishedWithDetails` (para página producto)
2. Remover `images` de queries de listado

**Tests Requeridos:**
- Test de tamaño de respuesta: < 5KB por producto en catálogo
- Test de integración: página detalle carga imágenes correctamente

---

### Prioridad 3: MEDIO - Mantenibilidad y Consistencia

#### ✅ Componente E: Exception Filters Centralizados
**Razón:** 15+ controladores con manejo de errores inconsistente.

**Cambios:**
1. Crear `app/api/src/common/filters/http-exception.filter.ts`
2. Crear `app/api/src/common/interceptors/success.interceptor.ts`
3. Registrar globalmente en `main.ts`

**Tests Requeridos:**
- Test de integración: endpoints devuelven formato de error estándar
- Test de logging: errores 5xx se loguean con requestId

#### ✅ Componente F: Extracción de Lógica Duplicada
**Razón:** 3+ servicios consultan tiendas por usuario.

**Cambios:**
1. Crear `StoreAccessService` con método `getUserStoreIds(userId)`
2. Inyectar en `DashboardService`, `DashboardUnifiedService`, etc.
3. Agregar cache Redis para resultado (TTL 5min)

**Tests Requeridos:**
- Test unitario: verificar cache hits/misses
- Test de integración: múltiples llamadas devuelven mismo resultado
